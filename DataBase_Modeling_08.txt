1. RELATIONSHIP 정의 절차
- 1. 존재여부의 결정
  = 1. RELATIONSHIP MATRIX 활용
  = 2. 심증 -> 직접 종속 판단 (접근경로 단축 여부 확인)
  = 3. Sample Data 활용
  = 4. 타 시스템 경험 활용 + 사용자에게 다양한 질문 (아니라는 결론도 중요)
  = 5. 현재의 관계 + 장래에 사용될 경우도 고려

- 2. 관계명칭 부여
  = 1. 물증 찾기 (관계 명을 RELATIONSHIP MATRIX 에 기록 - 1차 선정 관계 대상)
  = 2. 구체적인 관계 지정
  = 3. `~에 관계되어` & `~에 속하여` 라는 애매한 표현 및 일반적인 표현 삼가기
  = 4. 관계를 맺는 `구체적 대상`, `구체적 관계 내용` 부여
  = 5. 양방향 모두 지정이 원칙 (상황에 따라 한 방향만 정의 가능)
  = 6. 제1자 입장에서 정의 (상황에 따라 제3자 입장에서 정의)
  = 7. RELATIONSHIP READ 형식으로 문장 만들어 보기

2. RELATIONSHIP
- 1. 관계 형태 결정
  = 1. One or More (1 : M)
  = 2. M : M (매우 적은 가능성) -> 1 : M 관계로 하기 위한 업무규칙의 재정의 검토
  = 3. 경우에 따라 발생횟수를 조사
  = 4. 기존 시스템의 DATA 참조
  = 5. 양쪽 방향 모두 조사

- 2. 선택 사양 결정
  = 1. 일반적이고 상식적인 선에서 먼저 판단
  = 2. 그 관계를 만족해야만 하는지 파악
  = 3. 만족되지 않는 경우 확인 (있다면 MAYBE)
  = 4. 양쪽 방향 모두 조사
  = 5. 관계 형태에 따른 유형과 대비해 볼 것

- 3. 관계의 검증
  = 1. 업무의 일반적인 규칙과 대비
  = 2. READ RULE 에 따라 문장 작성
  = 3. 하위 Entity 추가 시 상위 Entity 관계 재조명
  = 4. 기존 시스템과 비교 분석

3. E-R Diagram 작성
- 1. 작성 원칙
  = 1. ENTITY BOX 들은 가능한 정렬
  = 2. 수직, 수평선 주로 사용
  = 3. 사선 사용 시, 30도 ~ 60도 내에서 사용
  = 4. 빈 공간 많이 사용하여 복잡해 보이지 않게 설정
  = 5. 라인 간 충분한 간격
  = 6. 모든 용어 중복 금지
  = 7. 약어나 특수 용어 사용 금지
  = 8. 문자는 수평선 맞추기
  = 9. 관계명칭은 각 방향 라인의 끝부분에 기술
  = 10. 모양과 배치에 나름의 특징 부여
  = 11. 격자형태의 지면에 작성 금지
  = 12. 전체적인 형태에 따라 ENTITY BOX 크기 조절
  = 13. 교차선 생기거나 관계선이 너무 길지 않도록 설정
  = 14. 필요에 따라 Key Entity 복제본을 사용해도 무방
  = 15. 복제본은 항상 주는 (참조되는) 쪽

- 2. 실전 연구 (RelationShip)
  = 1. 부모 자식간은 정말 1:M 인가?
  = 2. 관계란 규정하기에 따라 얼마든지 다양한 형태로 존재
  = 3. 집합정의에 따라 관계형태는 달라짐
  = 4. `사원 Entity` 와 `부서 Entity` 는 과연 1 : M 관계에 있는지에 대한 여부
  = 5. 실무에서는 이들 관계를 어떻게 정의해야 하는지에 대한 여부

  = 6. Example (사원과 부서)
     -> 1. 실전에서는 근무 이력은 특이한 경우에만 발생 -> 대게 데이터 발생 시점을 기준
     -> 2. 관련 부서를 찾기 위해서는 M : M 조인 발생
         => 조건을 주어서 하나의 부서만 찾을 수 있지만, 비효율 발생할 수 있음

- 3. 관계 (혹은 집합) 의 분할 형태
  = 1. 한 개로 가져감 (근무 이력 포함)
  = 2. 완전 분할 (조각 -> 1 : M, 전체 모양 -> M : M) => Relation 2개로 쪼개서 가져가기
  = 3. 중복 분할 (조각 따로, 전체 모양 따로 가져가기)
