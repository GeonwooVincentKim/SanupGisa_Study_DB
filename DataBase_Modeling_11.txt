1. ATTRIBUTE 의 지정
- 1. 지정 목록
  = 1. DFD 의 DATA STORE
  = 2. 현업부서의 각종 장표, 보고서
  = 3. 기존 시스템 참조
  = 4. FILE DUMP
  = 5. 타사 자료 참조
  = 6. 유사 시스템 참조
  = 7. 사용자에게 질문 및 확인
  = 8. 장래 사용예상 항목 감안

- 2. 지정 목록 점검
  = 1. 과연 언제 사용되는가
  = 2. 어떻게 사용되는가
  = 3. 정말 필요한가
  = 4. 추출 항목 (Derived Data) 이 아닌

2. 식별자 (UID) 지정 => 기본키
- 1. UID -> Unique IDentifier
- 2. 하나, 혹은 하나 이상의 ATTRIBUTE 로 구성
- 3. 모든 ENEITY 는 반드시 UID 를 가져야 함
- 4. UID 를 가지지 못하면 ENTITY 가 아님
- 5. UID 구성하는 모든 ATTRIBUTE 는 반드시 존재 (Mandatory)
- 6. UID Attribute 에는 `#*` 를 표시
- 7. 대부분의 Key Entity 는 하나의 ATTRIBUTE 로 구성
  = 1. `외` 자임
  = 2. 우리 성씨와 같다고 생각하면 됨
     -> Example => `김`씨, `박`씨, `이`씨, `최`씨 ..

- 8. Main Entity 와 Action Entity 는 의미상의 주어가 UID
- 9. 상황에 따라 의미상의 주어는 인조키 (얼굴마담) 로 대체 될 수 있음
- 10. 무조건 인조키 만들거나 상속받은 구성만으로 UID 구성하는 것은 아님 
- 11. 전략적인 판단 요구
- 12. 자신의 ATTRIBUTE 가 아니면서 RELOATION 을 위해 존재하게 되는 ATTRIBUTE 를 자신의 ATTRIBUTE 로 표시해선 안됨

2-1. 식별자 Tip
- 1. 모델링의 기본키 = 식별자
- 2. Table 에서의 기본키 = PK
- 3. 상속과 상속 간의 단절을 잘 활용하여 이름을 잘 이어야 한다
  = Example
     -> 1. 단일민족이라고 하지만 어디선가 `성`씨 (김, 박, 이, 최) 가 나뉘었고, 거기서도 또 어디선가 `파`가 나뉨 (김해 김씨, 안동 김씨)
     -> 2. 위와 같이 중간중간에 ENTITY 관계를 잘 단절시켜놓아야 관리하기 편리함

3. RELATIONSHIP 을 통한 UID 지정
- 1. UID BAR 는 UID 에 RELATIONSHIP 이 포함되어 있다는 의미
  = 즉, 계좌 ENTITY UID = 은행 코드 + 계좌번호

- 2. UID 는 Mandatory 여야 하므로, UID BAR 는 항상 `Mandatory` + `ONE and ONLY ONE ( ____ )` 관계여야 함
- 3. UID BAR 정의
  = 1. FK 이면서 PK 로 내려 갔다는 것
  = 2. 계좌와 은행
     -> 1. 은행코드 빼고, 계좌번호 가지고 UNIQUE 한다는 뜻
     -> 2. UID BAR 가 빠지면 은행코드가 계좌번호에 오지 않는 것이 아니라, 오긴 오 돼, UID 가 아니라는 것
     -> 3. `UID BAR` 로 인하여 `성`씨가 달라질 수 있음 (`김`씨에서 `박`씨로 변경할 수 있게 됨)
